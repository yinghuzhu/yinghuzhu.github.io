电商高并发库存超卖及用户限购解决方案

发表于 2022-12-18

1. 背景与挑战

电商促销活动（如秒杀）期间，系统会面临瞬时的高并发请求，对商品库存进行扣减。核心挑战包括：；

  • 库存超卖： 多个请求在极短时间内并发读取和修改库存，导致实际售出数量超过真实库存。
  • 用户限购： 确保特定活动下，每个用户对同一商品只能购买限定数量（例如1件）。
  • 高QPS： 需处理极高的并发请求（例如10万QPS）。
  • 数据一致性： Redis缓存与DB数据库之间的数据一致性问题。
  • 用户体验： 在高压下仍能提供快速响应和明确的反馈。

2. 核心设计原则

  • 层层过滤： 在请求到达核心逻辑前，尽可能过滤无效和恶意请求。
  • 异步削峰： 利用消息队列将核心写操作异步化，缓解DB压力。
  • 原子操作： 利用Redis Lua脚本保证关键操作（库存检查、限购判断、库存扣减、购买标记）的原子性。
  • 最终一致性： 接受缓存与DB在短时间内的不一致，通过补偿和对账机制保证最终一致。
  • 快速失败与反馈： 对不满足条件的请求快速拒绝，并给出明确提示。

4. 详细设计

4.1 前端优化

  • 按钮状态： 活动开始前按钮置灰；快速判断无库存或用户已购后，前端按钮置灰或提示。
  • 页面静态化： 商品详情页使用CDN加速，减少动态请求。
  • 验证码/答题（可选）： 增加简单门槛，过滤机器流量。

4.2 流量接入与过滤 (Nginx+Lua / API Gateway)

  • IP/UID限流： 基础限流，防止单一来源攻击。
  • WAF防护： 防DDoS、SQL注入、刷单等。
  • “售罄”快速拦截：
    • 在Redis中设置全局售罄标记：product_sold_out_flag:{product_id}。
    • 网关层可读取此标记，若已售罄，直接返回错误，避免请求进入后端。

4.3 应用服务层

  • 用户身份与活动资格校验：
    • 确保用户登录。
    • 快速判断用户是否已购买 (限购核心)：
      • 查询Redis Set：SISMEMBER user_purchase_set:{product_id} <user_id> (使用Hash Tag确保与库存key同slot)。
      • 如果已购买，直接拒绝。
  • 调用Redis Lua脚本处理核心逻辑。

4.4 核心库存与限购处理 (Redis Cluster + Lua)

4.4.1 Redis Key设计 (使用Hash Tag确保同Slot)

  • 商品库存Key： product_stock:{<product_id>} (String类型, 存储库存数)
  • 用户购买记录Key (限购)： user_purchase_set:{<product_id>} (Set类型, 存储已购买该商品的用户ID)
  • 全局售罄标志Key： product_sold_out_flag:{<product_id>} (String, 值为1表示售罄)

4.4.2 Lua脚本 (原子操作)

4.4.3 应用层调用Lua脚本逻辑

  1. 应用服务调用EVAL执行Lua脚本。
  2. 根据返回值处理：
    • 成功 (返回值 >= 0)：
      1. 立即生成一个“待处理”订单，并记录一个操作令牌/预订单ID到Redis（带TTL）。 例如：HSET pending_orders:<order_id> user_id <user_id> product_id <product_id> timestamp <ts>，并 EXPIRE pending_orders:<order_id> 300 (5分钟)。
      2. 将订单信息（订单ID、商品ID、用户ID等）发送到消息队列 (MQ)。
      3. 快速响应用户“下单成功，处理中”或“排队中”。
    • 失败 (返回值 < 0)： 直接返回用户对应错误提示（已购买、库存不足、商品问题、已售罄）。

4.5 异步持久化 (消息队列 + 订单处理服务)

  • 消息队列 (MQ)： Kafka, RocketMQ等。缓冲Redis处理成功的下单请求。
  • 订单处理服务 (消费者)：
    1. 从MQ拉取消息。
    2. 幂等性处理： 基于订单ID或消息中的唯一业务ID，防止重复消费（可在Redis中记录已处理ID）。
    3. 数据库事务操作：
      • 再次校验用户购买记录 (DB兜底)： 查询 user_purchase_log 表。若已存在，则为重复消息或异常，记录并忽略/标记。
      • 扣减数据库库存： UPDATE product_inventory SET stock = stock - 1 WHERE product_id = ? AND stock >= 1; 检查影响行数。
      • 持久化用户购买记录： INSERT INTO user_purchase_log (user_id, product_id, activity_id, order_id, ...) VALUES (...); (需有 UNIQ(user_id, product_id, activity_id) 唯一约束)。
      • 创建/更新订单状态为“已支付/待发货”等。
    4. 处理结果：
      • 事务成功： 提交事务。从Redis中删除之前步骤 4.4.3.1 中记录的 pending_orders:<order_id> 令牌。
      • 事务失败 (DB库存不足、唯一键冲突等)：
        • 回滚事务。
        • 执行补偿逻辑：
          • Redis库存回滚：INCRBY product_stock:{<product_id>} 1
          • Redis用户购买标记移除：SREM user_purchase_set:{<product_id>} <user_id>
          • 清除可能已设置的全局售罄标志 (如果是因为此订单失败导致库存不为0了)：检查DB库存，如果不为0，则 DEL product_sold_out_flag:{<product_id>}
        • 标记订单为失败，记录错误日志。
        • (可选) 通知用户订单创建失败。

4.6 异常处理与数据一致性保障

4.6.1 Redis Lua成功，后续应用崩溃 (生单/发MQ前)

  • 超时未处理订单回库存机制：
    • 一个定时任务/延迟队列扫描Redis中的 pending_orders:* (或其他“待处理”标记)。
    • 如果一个“待处理”标记（如 pending_orders:<order_id>）存在时间超过预设阈值（例如5-10分钟），且在DB中未找到对应成功创建的订单。
    • 则认为后续DB操作失败，执行补偿：回滚Redis库存 (INCRBY)，移除用户购买标记 (SREM)，删除 pending_orders:<order_id> 标记。
    • 这个机制是针对应用在将订单信息发送到MQ之前就崩溃的情况，或者MQ消息丢失的情况。

4.6.2 MQ消费者DB操作失败

  • 已在4.5中描述，消费者事务失败后需补偿Redis。

4.6.3 定期对账

  • 定期（如每日低峰期）执行脚本，校对Redis缓存库存/用户购买记录与DB中的数据。
  • 发现不一致时，根据业务规则（通常以DB为准，或结合操作日志）进行修复，并发出告警。

4.7 监控与告警

  • Nginx/Gateway： QPS、错误率、限流触发次数。
  • 应用服务： 接口响应时间、错误率、Redis连接池状态。
  • Redis Cluster： 内存使用、CPU、命中率、慢查询、主从同步状态、各节点负载均衡情况（关注Hash Tag可能导致的热点）。
  • 消息队列： 消息积压数、生产/消费速率。
  • 订单处理服务： 消费速率、DB操作耗时、错误率、补偿次数。
  • 数据库： 连接数、慢查询、磁盘I/O、CPU。
  • 业务指标： 库存数量、订单创建成功率、售罄商品列表。

5. Redis集群下多Key操作

  • 挑战： Lua脚本操作的多个Key (product_stock:{pid}, user_purchase_set:{pid}) 必须位于Redis Cluster的同一个slot。
  • 解决方案： 使用 Hash Tags。确保花括号{}内的字符串对于同一商品是相同的，例如：
    • product_stock:{<product_id>}
    • user_purchase_set:{<product_id>}
    • product_sold_out_flag:{<product_id>}
      这样，这些与特定商品ID相关的key会被路由到同一个Redis节点，Lua脚本可以原子地操作它们。
  • 潜在风险： 热点商品可能导致其对应的slot/节点成为瓶颈。需要监控节点负载，并考虑是否有超级热点商品需要特殊处理（如预热到本地缓存，或采用更复杂的库存分配策略，但这会增加系统复杂度）。

6. 总结

该方案通过多层协作，结合Redis Lua的原子性、消息队列的异步削峰以及必要的补偿和对账机制，旨在解决电商高并发场景下的库存超卖和用户限购问题。关键在于平衡性能、一致性和系统复杂度。在实施过程中，充分的压测、详细的日志记录和完善的监控告警是必不可少的。
