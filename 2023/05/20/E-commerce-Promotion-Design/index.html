电商促销规则系统技术方案

发表于 2023-05-20

1. 引言

1.1. 背景

随着电商业务的快速发展，灵活多样的促销活动成为提升销售额、吸引用户、清理库存、推广新品等关键运营手段。为了支持运营团队快速、高效地配置和管理各类促销规则，并保证促销活动的准确性、稳定性和高性能，特制定本电商促销规则系统技术方案 。

1.2. 目标

本方案旨在设计并实现一个通用、可扩展、高性能的电商促销引擎及其配套管理系统，实现以下核心目标：

  • 灵活性与可配置性： 支持运营人员通过可视化界面配置常见的促销规则（如满减、满折、打折、买赠、优惠券等）及其组合，无需代码开发。
  • 准确性： 保证促销优惠计算的精确无误，包括优惠叠加、互斥、分摊等复杂场景。
  • 高性能与高并发： 能够应对大促期间的高并发访问，保证促销计算不成为系统瓶颈。
  • 可扩展性： 系统架构易于扩展，能够快速支持未来新的促销类型和业务需求。
  • 可观测性与易管理性： 提供完善的活动管理、监控告警、效果分析等功能。

1.3. 范围

本方案主要涵盖促销规则引擎的核心设计、数据模型、系统架构、技术选型以及关键模块的实现思路。

2. 系统概述

电商促销规则系统主要由以下几个部分组成：

  1. 促销管理后台 (Promotion Admin Console): 供运营人员创建、配置、管理和监控促销活动。
  2. 促销引擎 (Promotion Engine): 核心模块，负责接收促销计算请求，根据配置的规则匹配并计算优惠。
  3. 规则库 (Rule Repository): 存储促销活动、规则、条件、动作等配置信息。
  4. 上下文构建器 (Context Builder): 负责收集和构建促销引擎所需的上下文信息（如购物车、用户信息）。
  5. 接口层 (API Layer): 提供对外接口，供购物车、订单结算、商品详情等业务系统调用。

3. 核心设计要点

3.1. 规则的抽象与配置化

  • 原子化条件 (Atomic Conditions): 将判断条件拆解为最小单元。
    • 例如：订单金额、商品数量、商品ID/品类/品牌、用户等级、用户标签、是否新用户、当前时间等。
  • 原子化动作 (Atomic Actions): 将优惠行为拆解为最小单元。
    • 例如：订单金额固定减、订单金额百分比折扣、商品固定减、商品百分比折扣、赠送指定商品、免运费、发放优惠券等。
  • 规则组合器 (Rule Composer): 允许将原子条件通过逻辑运算符（AND, OR, NOT）组合，并将满足条件的规则与一个或多个原子动作关联。

3.2. 上下文与事实 (Context & Facts)

促销引擎的输入是一个包含所有相关信息的“上下文”对象：

  • 购物车上下文 (CartContext):
    • 商品列表 (items): 每个商品包含 sku_id, quantity, original_price, category_id, brand_id, tags 等。
    • 当前购物车总金额 (total_amount)。
    • 当前购物车总件数 (total_quantity)。
    • 已选优惠券 (selected_coupons)。
  • 用户上下文 (UserContext):
    • 用户ID (user_id)。
    • 用户等级 (user_level)。
    • 用户标签 (user_tags)。
    • 是否新用户 (is_new_user)。
  • 环境上下文 (EnvironmentContext):
    • 当前时间 (current_time)。
    • 访问渠道 (channel: ‘APP’, ‘WEB’, ‘MINI_PROGRAM’)。

3.3. 规则匹配与执行

  1. 规则预加载与筛选：
    • 系统启动或活动变更时，预加载有效的促销活动和规则到内存缓存中。
    • 根据当前上下文（如时间、渠道、用户分群）初步筛选可能适用的活动。
  2. 条件评估：
    • 对于每个筛选出的活动规则，使用其定义的条件组（ConditionGroup）对当前上下文进行匹配。
    • 支持高效的条件匹配算法（可考虑使用轻量级规则引擎库或自研匹配逻辑）。
  3. 动作执行与优惠生成：
    • 对于匹配成功的规则，执行其关联的促销动作（PromotionAction）。
    • 生成优惠结果对象，包含优惠类型、优惠金额/描述、适用商品范围等。

3.4. 优先级、互斥与叠加

  • 优先级 (Priority): 为每个促销活动设置优先级，数字越大（或越小，需统一约定）优先级越高。高优先级规则先匹配和应用。
  • 互斥规则 (Exclusivity Rules):
    • 活动级别互斥： 设置某些活动（如秒杀）与其他所有或特定类型的活动互斥。
    • 互斥组 (Exclusive Group): 定义互斥组，同组内的促销活动仅能生效一个（通常是优先级最高的或优惠力度最大的）。
  • 叠加策略 (Stacking Strategy):
    • 明确定义优惠计算的顺序，例如：
      1. 单品级优惠（如商品直降、特定商品折扣）
      2. 商品组合优惠（如第二件半价，买M送N）
      3. 店铺级/订单级满减/满折
      4. 优惠券抵扣
      5. 运费计算与优惠
    • 系统需支持配置不同类型优惠是否可叠加。

3.5. 优惠计算与分摊

  • 精确计算： 金额计算使用高精度类型（如 Decimal 或以分为单位的整数）避免浮点数精度问题。
  • 优惠分摊 (Discount Allocation):
    • 订单级优惠（如满减、优惠券）需要按规则（如商品金额比例）分摊到订单内的各个商品上。
    • 分摊结果用于财务核算、退款处理、商家结算等。
    • 分摊记录需随订单信息持久化。

5. 系统架构

5.1. 模块划分

  • API层 (Gateway / BFF):
    • 接收前端（购物车、结算页）的促销计算请求。
    • 调用上下文构建器和促销引擎。
    • 返回优惠计算结果。
  • 促销管理后台 (Admin UI + Backend Service):
    • 提供促销活动、规则、优惠券等的CRUD操作界面。
    • 管理活动生命周期（上线、下线、暂停）。
    • 提供活动数据监控和报表。
  • 上下文构建器 (Context Builder Service):
    • 聚合来自商品服务、用户服务、购物车服务等的数据，构建促销引擎所需的完整上下文。
  • 促销引擎核心 (Promotion Engine Core Service):
    • 无状态服务，接收上下文和待匹配规则列表（或活动ID）。
    • 执行规则匹配、优先级处理、互斥判断、优惠计算。
    • 返回优惠结果列表。
  • 规则存储 (Rule Repository - Database):
    • 使用关系型数据库（如PostgreSQL, MySQL）或NoSQL数据库（如MongoDB，如果规则结构非常灵活）存储促销配置。
    • 关键表：PromotionActivity, PromotionRule, ConditionNode, PromotionAction, CouponTemplate, CouponInstance。
  • 规则缓存 (Rule Cache - In-memory Cache):
    • 使用Redis或Memcached等缓存已解析和预处理的活动规则，加速匹配。
    • 通过消息队列（如Kafka, RabbitMQ）订阅规则变更事件，实现缓存的近实时更新。

5.2. 技术选型建议

  • 后端语言框架： Java (Spring Boot), Go, Python (Django/Flask) - 根据团队熟悉度和性能要求选择。
  • 数据库： PostgreSQL (JSONB支持良好), MySQL。
  • 缓存： Redis。
  • 消息队列： Kafka, RabbitMQ (用于规则变更通知，异步任务等)。
  • 规则引擎 (可选): Drools (Java), 或者自研基于配置的轻量级匹配逻辑。对于特别复杂的场景，可考虑引入专用规则引擎。
  • 前端框架： React, Vue, Angular。
  • 服务间通信： gRPC, RESTful API。

5.3. 部署架构

  • 微服务化部署，每个核心模块作为独立服务。
  • 使用容器化技术（Docker, Kubernetes）进行部署和管理，便于伸缩。
  • 配置负载均衡器应对高并发。

6. 关键流程

6.1. 促销活动创建与配置流程

  1. 运营人员在管理后台选择促销类型（如满减）。
  2. 填写活动基本信息（名称、时间、优先级、互斥规则等）。
  3. 配置触发条件（如“购物车中指定分类商品满200元”）：
    • 选择条件类型（如“商品小计”）。
    • 设置条件参数（如“品类ID列表”，金额“200”）。
    • 组合多个条件（AND/OR）。
  4. 配置优惠动作（如“订单金额减50元”）：
    • 选择动作类型（如“订单固定金额减免”）。
    • 设置动作参数（如金额“50”）。
  5. 保存活动，系统校验规则配置的合法性。
  6. 活动状态变更为“草稿”或“待上线”。
  7. 运营人员可预览或测试活动效果。
  8. 活动到期或手动上线后，状态变为“已激活”，规则数据同步到缓存。

6.2. 购物车/结算页促销计算流程

  1. 用户在购物车或结算页操作（增删商品、选择优惠券）。
  2. 前端应用调用API层，传递当前购物车快照和用户信息。
  3. API层调用上下文构建器，生成完整的 PromotionContext。
  4. API层（或促销引擎网关）从规则缓存中获取当前有效的、可能适用于该用户的 PromotionActivity 列表。
  5. API层将 PromotionContext 和筛选后的活动规则列表传递给促销引擎核心。
  6. 促销引擎核心：
    • 遍历活动规则： 按优先级处理。
    • 条件匹配： 对每个规则的条件树进行评估。
    • 互斥与叠加处理： 根据配置决定哪些规则可以同时生效。
    • 优惠计算： 执行匹配成功的规则的动作，计算优惠金额或生成赠品信息。
    • 优惠券处理： 如果用户选择了优惠券，验证其可用性并计算抵扣。
    • 构建优惠结果： 生成包含所有适用优惠的列表。
  7. 促销引擎核心返回优惠结果列表给API层。
  8. API层将优惠结果返回给前端展示。
  9. 用户提交订单时，选定的优惠信息（AppliedPromotion）随订单一同保存。

6.3. 规则变更与缓存更新

  1. 运营人员在后台修改或上下线促销活动。
  2. 促销管理后台服务更新数据库中的规则。
  3. 通过消息队列（如Kafka）发布规则变更事件。
  4. 促销引擎实例订阅该事件，从数据库重新加载变更的规则，并更新本地缓存。
    • 或者，缓存设置较短的过期时间，定期从数据库拉取最新规则。

7. 性能与扩展性考虑

  • 缓存策略： 大量使用缓存（规则配置、热点商品属性、用户标签等）。
  • 异步处理： 对于非核心路径的优惠计算（如“猜你喜欢”处的优惠提示）可考虑异步。
  • 规则预编译/解析： 将配置化的规则在加载时转换为更易执行的内部表示。
  • 水平扩展： 促销引擎核心服务设计为无状态，便于水平扩展实例数。
  • 数据库优化： 对规则查询进行索引优化。
  • 代码层面优化： 避免在规则匹配循环中进行昂贵的I/O操作或复杂计算。

8. 安全与风控

  • 输入校验： 对所有来自前端和管理后台的输入进行严格校验。
  • 权限控制： 管理后台操作需进行身份认证和权限校验。
  • 防刷单机制：
    • 限制单用户参与活动的次数/频率。
    • 监控异常IP、设备。
    • 对高价值优惠活动增加验证码等手段。
  • 数据安全： 敏感数据加密存储。

9. 监控与告警

  • 业务指标监控： 活动参与率、优惠券核销率、促销带来的GMV提升等。
  • 系统指标监控：
    • 促销引擎接口QPS、平均响应时间、错误率。
    • 规则缓存命中率。
    • 数据库连接池、CPU、内存使用率。
  • 告警机制： 关键指标超过阈值时，通过邮件、短信等方式及时告警。
  • 日志记录： 详细记录规则匹配过程、优惠计算步骤、错误信息，便于问题排查。

10. 迭代计划 (示例)

  • Phase 1 (MVP):
    • 支持核心促销类型：满减、满折、单品直降、优惠券（固定金额、满减）。
    • 基础的规则配置后台。
    • 核心促销引擎实现，支持优先级和简单互斥。
    • 基础的缓存机制。
  • Phase 2:
    • 支持更多促销类型：买赠、第N件优惠、打包价、预售。
    • 完善的优惠叠加和互斥逻辑配置。
    • 优惠券模板化管理。
    • 初步的活动效果数据统计。
  • Phase 3:
    • 引入DSL或可视化规则编排器，提升规则配置灵活性。
    • 精细化用户分群与定向促销。
    • 与风控系统集成。
    • 完善的监控告警和性能优化。
  • 后续： A/B测试平台集成、智能推荐促销等。

11. 总结

本方案提供了一个相对全面的电商促销规则系统设计思路。在实际建设过程中，需要根据具体业务需求、团队能力和现有技术栈进行调整和细化。核心在于构建一个灵活配置、准确计算、高性能、易扩展的促销引擎，以支撑电商业务的持续增长和创新。
